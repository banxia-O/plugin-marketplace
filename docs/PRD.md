# 插件百宝阁

> 给你的 AI 找趁手的工具

**域名**：`plugin.md-banxia.cn`
**定位**：国内（中国）面向非技术用户的 AI Agent 插件精选市场
**核心价值**：解决国内用户"不知道有什么插件、不会装、不知道好不好用"的信息差痛点
**前置条件**：用户需拥有一个可运行的 AI Agent（没有 agent 装不了插件）

---

## 一、目标用户

| 角色 | 画像 | 核心需求 |
|------|------|----------|
| 普通用户（User） | 非程序员，有 agent 但不懂技术，初中生/高中生也能用 | 无脑找到好用的插件，给 agent 一个链接就能跑 |
| Agent | 各平台的 AI 智能体（Claude、OpenClaw、Hermes、Kimi、通义、扣子等） | 读 agent.md 就知道插件是什么、怎么装、怎么验证、怎么向用户收集种子数据 |
| 插件开发者 / 搬运者 | 写了 MCP 插件的开发者，或收集到优质插件的用户 | 最少操作完成上架，获得曝光和 credits |

### 2.4 agent.md 规范

每个插件必须有 agent.md，这是本平台的核心差异化产品。

**核心理念：user 只需要给 agent 一个 GitHub 链接，agent 读完 agent.md 后能全流程跑通——安装、配置、验证、引导用户提供种子数据。**

**agent.md 模板结构**：

```markdown
# [插件名称]

## 部署方式
本地部署 / 远程服务 / 两者皆可

## 这个插件是什么
用一段话告诉 agent 和 user，这个插件能做什么，解决什么问题。

## 适用场景
列出 2-5 个典型使用场景，让 user 判断"这个跟我有没有关系"。

## 安装与配置

### 环境要求
需要什么环境（Node.js / Python / Docker 等），版本要求。

### 安装步骤
完整的命令行操作步骤，面向零基础用户：
1. clone 仓库
2. 安装依赖
3. 启动服务
4. 连接到 agent

（如为远程服务，则写接入 URL 和认证方式）

### 配置项
所有需要配置的参数，以 JSON/YAML/环境变量 形式给出，可直接复制使用。
含默认值和每个字段的说明。

### 模型配置（如需要）
如果该插件需要配置 LLM：
- agent 应自检当前系统已配置的 API Key
- 列出支持的模型提供商（OpenAI / DeepSeek / 通义 / 本地模型等）
- 引导 user 选择使用哪个模型（有什么模型和选什么模型是 user 的事）

## 验证
安装完成后，agent 应执行以下操作验证插件是否正常工作：
- 具体的测试命令或测试用例
- 预期的成功输出
- 常见失败情况及排查方法

## 需要 user 提供的种子数据
列出 agent 启动这个插件前需要向 user 收集的冷启动数据。
例如：API Key、文件路径、账号信息、工作目录等。
对每项数据说明：是什么、为什么需要、从哪里获取。

## 使用示例
展示 agent 和 user 的典型对话流程，让 agent 知道怎么向 user 报告这个插件的价值。

## 注意事项
权限、安全、费用等需要提前告知 user 的内容。
```

**生成规则**：

1. 源仓库自带 agent.md → 审核质量后直接使用
2. 源仓库无 agent.md，README 信息充分 → 工作流基于 README 自动生成（中文），标注"AI 生成，欢迎贡献改进"
3. README 信息不足（<200字 / 结构不清晰）→ 标记"⚠️ agent.md 待完善"，不强制生成，不消耗额外 token

### 2.5 搜索

- V1 实现中文模糊搜索（关键词匹配插件名称 + 描述 + 分类标签）
- 搜索框常驻顶部
- 支持大白话搜索："翻译""微信""表格""写论文""医学文献"

### 2.6 用户系统

#### User 注册/登录

V1 支持：
- 用户名 + 密码注册（最低门槛）
- 邮箱注册（支持 QQ邮箱、163邮箱、Gmail 等主流邮箱）
- GitHub 账号登录（OAuth）

V2 增加：
- QQ 登录
- 微信登录
- 手机号 + 验证码

#### Agent 注册/登录

Agent 拥有独立账户体系：
- 账号 + 密码注册/登录（适合手动配置场景）
- API Token 鉴权（适合自动化工作流场景，agent 无需每次输入密码）
- 两种方式对应同一个 agent 账户和同一个 credits 余额
- Agent 可独立浏览、下载插件

#### User-Agent 绑定

- User 可在个人设置中绑定 Agent（填写 Agent 账号）
- Agent 端也可引导 User 绑定（反向绑定）
- 一个 User 可绑定多个 Agent，一个 Agent 可绑定一个 User
- 绑定后 credits 可双向转账，实现积分共享
- Agent 的下载行为关联到 User 账户（可追溯）

## 三、Credits 经济系统（V2）

> V1 暂不实现，但后台记录所有行为数据（上传、下载、点赞），V2 上线后按规则补发。
> V1 上传页面明确告知："当前为公测期，所有贡献行为已记录，积分系统上线后将按规则补发。"

| 行为 | Credits 变动 |
|------|-------------|
| 新用户注册 | +500 |
| 分享到社交媒体并验证 | +1000 |
| 上传插件（审核通过） | +100 |
| 插件被下载（每次） | 上传者 +2 |
| 插件被点赞（每次） | 上传者 +5 |
| 下载插件 | -10（或按插件定价） |
| User-Agent 转账 | 双向，自由金额 |

**择优机制**：

- 下载量 + 好评数达到阈值 → 自动推荐到首页热门
- 长期高分插件 → 标记"精选"徽章
- 长期无更新 + 差评 → 降权或标记"已过时"

**薅羊毛利用策略**：

- 社交媒体验证即传播，每个"薅羊毛"的用户都是免费推广员
- 前期宽松，后期根据数据收紧规则（如限制每日注册量、加人机验证）

**V1 隐藏账本**：

- 后台数据库记录所有上传/下载/点赞行为及时间戳
- 前端不展示积分，但数据完整保留
- V2 上线时按记录批量补发，公测期贡献者不吃亏

## 四、插件上传流程

### User 需要做的（最少操作）

1. 填写插件名称
2. 填写一句话描述
3. 选择功能分类（多选）
4. 填写 GitHub 仓库地址
5. 原作者信息：由 GitHub API 自动抓取仓库 owner 并填充，允许手动修改（应对 fork 仓库等情况）
6. 上传者信息：自动匹配当前登录用户名（与 credit 账号直接挂钩）
7. 勾选协议："我确认该仓库为开源项目（MIT / Apache 2.0 等宽松许可证），并已注明原作者信息"

提交 → 完。

### 平台自动处理（工作流）

```
用户提交
  ↓
工作流 Step 0：查重
  - 检测该 GitHub 仓库地址是否已存在于数据库
  - 已存在 → 拒绝，返回"该插件已上架"并附链接
  - 防止重复上传 / 刷 credits
  ↓
工作流 Step 1：验证 GitHub 仓库
  - 仓库是否存在、是否 public
  - 许可证类型检测（拒绝 GPL/AGPL）
  - README 是否存在
  - 通过 GitHub API 抓取原作者信息（owner 字段）
  ↓
工作流 Step 2：安全扫描
  - 基础代码扫描（依赖项检查、已知漏洞）
  - LLM 审核（读代码判断是否有明显恶意行为）
  - 标注审核状态：✅ 已审核 / ⚠️ 基础审核 / ❌ 未通过
  ↓
工作流 Step 3：内容生成
  - 检测仓库是否自带 agent.md → 有则审核后直接使用
  - 无 agent.md：
    - README 信息充分（≥200字且结构清晰）→ 基于 README 生成 agent.md（中文）
    - README 信息不足 → 标记"⚠️ agent.md 待完善"，不生成，不烧 token
  - 提取完整描述
  - 推断部署方式（本地 / 远程 / 两者）
  - 检测支持平台（从代码结构推断）
  ↓
工作流 Step 4：上架
  - 生成插件卡片
  - 插件进入"新上架"区域
  - 通知上传者审核结果
  - 后台记录上传行为（V1 隐藏账本）
```

## 五、安全策略

**审核声明**（页面底部常驻）：

> 本平台采用 AI 自动审核，覆盖基础安全检查（许可证合规、已知漏洞扫描、代码行为分析）。审核不构成安全保证，使用插件前请自行评估风险。

**具体措施**：

- 仅接受宽松许可证（MIT、Apache 2.0、BSD、ISC）
- 同仓库地址查重，防止重复上传和 credits 刷取
- 依赖项 CVE 扫描
- LLM 代码审核（检测可疑网络请求、数据外传、文件系统越权访问）
- 用户举报机制（V2）
- 恶意插件发现后立即下架 + 封禁上传者
